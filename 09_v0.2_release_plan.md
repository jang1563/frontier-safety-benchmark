# v0.2 Release Plan

Date: 2026-04-01
Project: CBRN-AI 2.0 - Frontier Uplift Observatory
Release target: v0.2.0
Recommended window: 4-6 weeks
Scope: Public-facing, safety-oriented benchmark expansion and tooling hardening

## Executive summary

`v0.1` proved that the project can support:

- a safe public benchmark concept
- versioned schemas
- validated reviewed-response files
- a basic mitigation-audit workflow
- release and governance documentation

`v0.2` should be the release where the project becomes meaningfully broader, better documented, and more credible as a portfolio artifact. It should still remain clearly public-safe and non-operational, but it should feel less like a scaffold and more like the first real benchmark program milestone.

The central theme of `v0.2` is:

- more breadth
- clearer taxonomy
- stronger documentation
- better release automation
- more persuasive demo outputs

## 1. What v0.2 should accomplish

By the end of `v0.2`, an external reviewer should be able to say:

- this is a real benchmark project, not just a concept note
- the public layer has enough structure and volume to demonstrate seriousness
- the release process is disciplined and repeatable
- the tooling is beginning to support multi-run comparisons cleanly
- the work is still strongly aligned with frontier-safety evaluation rather than drifting into unsafe specificity

## 2. Release thesis

`v0.2` is the "credibility release."

`v0.1` established the bones of the project.
`v0.2` should establish credibility in four ways:

- benchmark coverage becomes broader and more balanced
- taxonomy becomes explicit and documented
- release and annotation process become easier to follow
- the results layer becomes more compelling for portfolio use

## 3. Scope

### In scope

- expand the safe public benchmark set
- create a formal taxonomy document
- create an annotation handbook
- add contamination-policy documentation
- add release metadata and machine-readable manifests
- expand the audit tooling to support clearer multi-run analysis
- produce a more polished demo result package

### Out of scope

- populating restricted content
- publishing operational examples
- human-subject studies
- external advisory board process
- full website deployment
- full platformization or dashboard infrastructure

## 4. v0.2 goals

### Goal 1. Expand benchmark breadth

Move from a tiny starter set to a small but credible public benchmark slice.

Recommended targets:

- 24-36 public items total
- at least 8 items in `public_dev`
- at least 12-20 items in `public_eval`
- at least 4 domain families represented
- at least 5 reasoning types represented

The objective is not scale for its own sake. The objective is enough breadth to make the benchmark card and release notes feel real.

### Goal 2. Make taxonomy explicit

The current taxonomy exists in schemas and notes, but not as a first-class public artifact. `v0.2` should turn it into a documented part of the benchmark.

Deliverables:

- taxonomy definitions
- label semantics
- examples of category boundaries
- documentation for `public_release_status` vs `evaluation_split`

### Goal 3. Strengthen annotation and review process

`v0.1` has reviewer concepts, but they are not yet an operational handbook. `v0.2` should make annotation and review replicable.

Deliverables:

- annotation handbook
- reviewer checklist
- scoring guidance for each metric
- error-tagging rules with examples

### Goal 4. Improve release hygiene

`v0.2` should move from informal release bookkeeping to a more machine-readable structure.

Deliverables:

- release manifest
- dataset manifest
- benchmark version metadata
- run metadata for demo outputs

### Goal 5. Improve results quality

The current audit output is useful but minimal. `v0.2` should produce more persuasive artifacts for portfolio and evaluation review.

Deliverables:

- cleaner markdown result reports
- machine-readable scorecards
- simple visual summaries or table-ready outputs
- comparison-ready run structure

## 6. Data design targets

The public benchmark set in `v0.2` should feel balanced rather than repetitive.

### Domain-family coverage target

Recommended minimum distribution:

- `refusal_and_safe_completion`: 5-7 items
- `mitigation_sensitive_query_handling`: 5-7 items
- `scientific_interpretation`: 5-7 items
- `evaluation_governance`: 4-6 items
- `policy_sensitive_response_handling`: 3-5 items

### Reasoning-type coverage target

Recommended minimum distribution:

- `safe_boundary_articulation`
- `factual_uncertainty_handling`
- `comparative_reasoning`
- `evidence_synthesis`
- `paraphrase_robustness`
- `missing_information_recognition`

### Balance rules

Try to avoid:

- too many near-duplicate boundary items
- too much concentration in one domain family
- all high-mitigation items and no medium-mitigation items
- only "easy" items with obvious ideal responses

## 7. Tooling goals for v0.2

### Audit script improvements

Extend the evaluator to support:

- CSV export
- deterministic ordering of outputs
- optional filtering by `model_name`
- optional filtering by split
- taxonomy-aware coverage summary

### Validation improvements

Add validation for:

- run manifests
- release manifests
- item-count coverage targets
- taxonomy coverage warnings

Warnings are appropriate for coverage balance; hard failures are appropriate for schema and duplication errors.

## 8. Documentation goals for v0.2

### Taxonomy doc

The taxonomy document should include:

- definitions for each domain family
- definitions for each reasoning type
- examples of ambiguous boundaries
- guidance on when to split or merge categories

### Annotation handbook

The handbook should include:

- scoring criteria for each metric
- examples of strong vs weak responses
- how to assign error tags
- what to do with borderline cases

### Contamination policy

The contamination doc should include:

- how provenance is recorded
- how refresh items are selected
- how public discussion affects benchmark freshness
- when an item should be retired or replaced

## 11. Acceptance criteria

`v0.2` should not be considered done unless all of the following are true:

### Data

- at least 24 public items exist
- all items validate against schema
- all reviewed responses validate against schema
- each reviewed model has complete item coverage for the released evaluation set

### Documentation

- taxonomy doc exists
- annotation handbook exists
- contamination policy exists
- benchmark card is updated for `v0.2`
- release log is accurate and complete

### Tooling

- validator passes
- evaluator runs cleanly
- results are written in markdown and JSON
- at least one structured machine-readable score output exists

### Portfolio value

- there is a benchmark card you would be comfortable linking
- there is a result summary you would be comfortable showing in an interview
- there is a clear story for what improved from `v0.1` to `v0.2`

## 12. Non-goals for v0.2

To keep scope sane, explicitly avoid:

- adding restricted benchmark items
- building a web app
- building a full dashboard
- adding complex automated judging
- trying to solve external validation in the same release

`v0.2` should be about structure, breadth, and clarity, not maximal feature count.

## 13. Main risks

### Risk 1. Quantity without quality

If you add many items too fast, the benchmark becomes repetitive and weak.

Mitigation:

- use coverage targets and redundancy review
- prefer 24 strong items over 40 weak ones

### Risk 2. Docs drift from code

As files multiply, it becomes easier for docs to describe features that do not exist.

Mitigation:

- update release log only after scripts run
- add manifest validation
- keep README scoped to actual artifacts

### Risk 3. Taxonomy bloat

Too many categories will make annotation harder and the public artifact less legible.

Mitigation:

- keep the taxonomy compact
- add examples rather than more labels when possible

### Risk 4. Portfolio abstraction

If the release becomes too documentation-heavy, it may still feel abstract.

Mitigation:

- include concrete result outputs
- include coverage summaries
- include a benchmark card with explicit version delta from `v0.1`

## 14. Definition of done

The release is done when:

- the benchmark is visibly larger and better balanced
- the taxonomy is explicit
- the annotation process is documented
- the release is reproducible and validated
- the result package is presentable to an external reviewer

## 15. Recommended build order

Do the work in this order:

1. expand public items

3. write taxonomy doc
4. write annotation handbook
5. add run/release manifest schemas
6. extend scripts for scorecards and coverage

8. finalize benchmark card and release log
